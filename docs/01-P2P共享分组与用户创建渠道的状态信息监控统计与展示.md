# P2P共享分组与用户渠道：状态监控、统计与智能分析设计文档

## 1. 背景与目标

### 1.1 业务背景

随着 NewAPI 平台中 P2P 共享分组和用户自定义渠道功能的引入，渠道数量和复杂性显著增加。平台需要一套强大的监控和统计机制，以确保服务的稳定性、透明度及智能化运营。当前系统缺少对渠道健康度、性能、模型质量波动的精细化追踪，也无法为P2P共享分组提供聚合视图，这限制了管理员的洞察能力和用户的服务体验。

### 1.2 核心业务目标

本项目旨在构建一个三位一体的监控、统计与智能分析系统，实现以下核心目标：

1.  **渠道状态与性能的全景监控 (Channel Statistics)**:
    *   **目标**: 为每个渠道（无论是平台自营、用户创建还是P2P共享）提供一个全面的健康和性能仪表盘。
    *   **关键指标**: 覆盖延迟、失败率、缓存命中率、流量（TPM/RPM）、配额消耗、并发和活跃用户数等维度。
    *   **价值**: 使管理员能够快速识别性能瓶颈、预测成本、评估渠道质量，并为智能路由提供数据支持。

2.  **模型智能与特征波动监控 (Channel Intelligence Monitoring)**:
    *   **目标**: 建立一套自动化机制，用于检测和量化渠道背后AI模型的“降智”或“漂移”现象。
    *   **核心能力**:
        *   **基准管理**: 允许管理员为每个模型指定一个“基准渠道”，并生成该模型在编码、推理、风格等维度的行为指纹。
        *   **差异对比**: 定期对其他渠道的同一模型进行探测，并将其行为与基准进行量化对比、打分。
    *   **价值**: 为渠道的“质量”提供一个客观、可量化的评估标准，帮助用户选择高质量渠道，也为平台的优胜劣汰提供依据。

3.  **P2P共享分组的聚合统计 (P2P Group Statistics)**:
    *   **目标**: 为 P2P 共享分组提供一个聚合的性能与使用情况视图。
    *   **核心能力**: 将分组内所有渠道的同类指标（如TPM、RPM、失败率）进行逻辑汇总，形成分组的整体画像。
    *   **价值**: 帮助分组所有者和成员宏观地了解分组的整体服务能力、成本和健康状况。

4.  **高性能与低侵入性架构**:
    *   **目标**: 确保监控统计系统本身不会对核心的请求转发链路造成性能影响。
    *   **设计原则**: 采用**内存 -> Redis -> 数据库**的三级缓存架构，实现数据面与统计分析的异步解耦。数据采集在内存中瞬时完成，批量、错峰地同步到持久化存储。

## 2. 数据库设计

为了支持上述功能，需要对现有数据模型进行扩展，并增加新的统计和监控相关的表。

### 2.1 核心设计思想

*   **分离热冷数据**:
    *   **热数据 (Hot Data)**: 实时、高频更新的统计指标（如TPM、RPM、并发数）将存储在 `Redis` 和内存中，保证数据面性能。
    *   **冷数据 (Cold Data)**: 定期（如每15分钟）从Redis同步到数据库，用于长期存储、趋势分析和仪表盘展示。
*   **扩展 `channels` 表**: 直接在 `channels` 表中增加统计字段，用于存储**渠道级别**的聚合统计信息。
*   **新增 `group_stats` 表**: 用于存储 **P2P共享分组级别** 的聚合统计信息。
*   **新增模型监控相关表**:
    *   `model_baselines`: 存储管理员为每个模型设定的行为基准。
    *   `model_monitoring_results`: 记录每次模型质量探测的结果和与基准的差异得分。

### 2.2 表结构定义

#### 2.2.1 `channels` 表扩展

在现有的 `channels` 表中增加以下字段，用于存储渠道的统计信息和监控策略。

```sql
ALTER TABLE `channels`
ADD COLUMN `avg_response_time` INT DEFAULT 0,              -- 平均"首字"响应时间 (ms)
ADD COLUMN `fail_rate` DOUBLE PRECISION DEFAULT 0.0,       -- 请求失败率 (%)
ADD COLUMN `avg_cache_hit_rate` DOUBLE PRECISION DEFAULT 0.0, -- 平均缓存命中率 (%)
ADD COLUMN `stream_req_ratio` DOUBLE PRECISION DEFAULT 0.0,   -- 流式请求占比 (%)
ADD COLUMN `tpm` INT DEFAULT 0,                            -- 每分钟处理的Tokens数量
ADD COLUMN `rpm` INT DEFAULT 0,                            -- 每分钟请求数
ADD COLUMN `quota_pm` BIGINT DEFAULT 0,                    -- 每分钟消耗的额度
ADD COLUMN `total_sessions` BIGINT DEFAULT 0,              -- 区间总服务session数
ADD COLUMN `downtime_percentage` DOUBLE PRECISION DEFAULT 0.0, -- 区间停止服务时间占比 (%)
ADD COLUMN `unique_users` INT DEFAULT 0,                   -- 区间服务用户数 (去重)
ADD COLUMN `monitoring_config` TEXT;                       -- 模型智能监控策略 (JSON)
```

*   **`monitoring_config` JSON 结构示例**:
    ```json
    {
      "enabled": true,
      "target_model": "gpt-4",
      "test_interval_minutes": 60,
      "test_type": ["encoding", "style"],
      "evaluation_standard": "standard"
    }
    ```

用于存储P2P共享分组的聚合统计数据。


## 3. 渠道统计与性能监控 (Channel Statistics)

本章节详细设计渠道状态与性能的监控方案，以满足需求 1.1, 1.2, 1.3。

### 3.1 数据采集架构

为实现高性能与低侵入性，采用**内存 -> Redis -> 数据库**的三级数据流架构。

```mermaid
graph TD
    subgraph DataPlane [数据面 (Relay 节点)]
        Req[API请求] -->|"更新统计"| IM[内存计数器<br>(channel_id -> stats)]
    end

    subgraph Aggregation [聚合层]
        IM -->|"每分钟, gopool"| Redis[Redis 缓存<br>(Hash: channel_stats:{channel_id})<br>(ZSet: dirty_channels)]
    end

    subgraph Persistence [持久化与分析层]
        Redis -->|"每15分钟, 后台任务"| DB[数据库 (channels 表)]
    end

    style IM fill:#f9f,stroke:#333,stroke-width:2px
    style Redis fill:#ff9,stroke:#333,stroke-width:2px
    style DB fill:#9cf,stroke:#333,stroke-width:2px
```

1.  **L1 - 内存 (In-Memory)**:
    *   **位置**: `relay` 模块的 `postConsumeQuota` 函数。
    *   **动作**: 请求结束后，在**内存中**原子地更新一个 `sync.Map`，其键为 `channel_id`，值为该渠道的实时统计对象（包含请求数、token数、延迟等）。此操作极快，对请求延迟影响可忽略。

2.  **L2 - Redis (Cache & Buffer)**:
    *   **触发**: 后台 `gopool` 任务每分钟执行一次。
    *   **动作**:
        1.  遍历内存中的 `sync.Map`，将有更新的渠道统计数据**原子地**（`HINCRBY`, `ZADD`）累加到 Redis 的 `HASH` 结构中。
            *   `Key`: `channel_stats:{channel_id}:{model_name}`
            *   `Fields`: `total_requests`, `fail_count`, `total_tokens`, `total_quota`, `total_latency`...
        2.  同时，将发生过更新的 `channel_id` 添加到一个 `ZSET` (`dirty_channels`) 中，`score` 为当前时间戳。这用于标记哪些渠道的数据是“脏”的，需要同步到数据库。
    *   **淘汰机制**: Redis 中的 `channel_stats:*` 键设置合理的TTL（如7天），确保冷渠道的数据自动过期。

3.  **L3 - 数据库 (Database)**:
    *   **触发**: 独立的后台定时任务每 **15 分钟**执行一次。
    *   **动作**:
        1.  从 `dirty_channels` 中查询过去15分钟内有过更新的 `channel_id`。
        2.  **分批、错位**地从 Redis 读取这些渠道的聚合统计数据。
        3.  计算最终的统计指标（如 `fail_rate`, `avg_response_time`），并批量更新到 `channels` 表对应的统计字段中。
        4.  更新成功后，从 Redis 中清理已同步的 `channel_stats` 计数（或将其值重置为0）。

### 3.2 指标定义与计算

| 指标 | Redis 存储 | 数据库计算/存储 (`channels`表) |
| :--- | :--- | :--- |
| **平均首字响应时间** | `total_latency` (总和), `latency_count` (次数) | `avg_response_time = total_latency / latency_count` |
| **请求失败率** | `fail_count`, `total_requests` | `fail_rate = (fail_count / total_requests) * 100` |
| **平均缓存率** | `cache_hit_count`, `total_requests` | `avg_cache_hit_rate = (cache_hit_count / total_requests) * 100` |
| **流式请求占比** | `stream_req_count`, `total_requests` | `stream_req_ratio = (stream_req_count / total_requests) * 100` |
| **TPM/RPM** | `total_tokens`, `total_requests` | `tpm` / `rpm` (基于时间窗口计算) |
| **额度PM** | `total_quota` | `quota_pm` (基于时间窗口计算) |
| **区间总服务session数** | `session_set` (HyperLogLog 或 Set) | `total_sessions` (从 Redis HLL/Set 中 `PFCOUNT`/`SCARD`) |
| **区间停止服务时间** | `downtime_seconds` (禁用时累加) | `downtime_percentage` (与总时长计算) |
| **区间服务用户数** | `user_set` (HyperLogLog) | `unique_users` (从 Redis HLL 中 `PFCOUNT`) |

### 3.3 API 设计

新增一个管理API端点，用于前端展示渠道的详细统计信息。

*   **Endpoint**: `GET /api/channels/:id/stats`
*   **权限**: 管理员
*   **参数**:
    *   `period` (Query, string, optional): 时间区间，可选值为 `1h`, `6h`, `7d`, `30d` (默认 `1h`)。
*   **响应**:
    *   **逻辑**: 优先从 Redis 中读取指定时间区间的实时统计数据；如果 Redis 数据不完整，则从数据库 `channels` 表中读取最新的聚合统计作为补充。
    *   **数据结构**:
        ```json
        {
          "channel_id": 123,
          "period": "1h",
          "stats": {
            "avg_response_time": 350,
            "fail_rate": 1.2,
            "avg_cache_hit_rate": 25.5,
            "stream_req_ratio": 80.0,
            "tpm": 150000,
            "rpm": 30,
            "quota_pm": 75000,
            "total_tokens_in_period": 9000000,
            "total_quota_in_period": 4500000,
            "avg_concurrency": 2.5,
            "total_sessions": 150,
            "downtime_percentage": 0.5,
            "unique_users": 25
          }
        }
        ```

## 4. 模型智能与特征波动监控 (Channel Intelligence)

本章节详细设计模型智能与特征波动监控方案，以满足需求 2.1, 2.2, 2.3。

### 4.1 核心工作流

该功能的核心是**基准对比法**。管理员先选定一个可靠的渠道作为“黄金标准”，生成该渠道下特定模型的行为“基准”，然后系统定期使用相同的探测 prompt 去请求其他渠道的同一个模型，并将输出与基准对比，从而量化其“降智”或“漂移”程度。

```mermaid
graph TD
    subgraph Admin_Panel [管理后台]
        A[选择模型 gpt-4] --> B{选择基准渠道: Channel-A};
        B --> C[生成基准];
        C --> DB1(model_baselines 表);
    end

    subgraph Backend_Task [后台定时任务]
        DB1 --> D[读取基准];
        D --> E[遍历所有支持gpt-4的渠道];
        E --> F{对Channel-B执行探测};
        F --> G[获取输出];
        G --> H{与基准对比};
        H --> I[计算差异分];
        I --> DB2(model_monitoring_results 表);
    end

    subgraph Judge_LLM [裁判LLM (可选)]
        H -- "复杂场景" --> J[请求Judgement];
        J --> H;
    end
```

### 4.2 基准管理 (Baseline Management)

1.  **基准生成**:
    *   **触发**: 管理员在后台界面为某个模型（如 `gpt-4-turbo`）选择一个基准渠道（如 `channel-123`），并选择检测类型（如 `style`）和评估标准（如 `strict`）。
    *   **动作**:
        1.  系统从预定义的 prompt 库（`@docs/001-具体检测AI降智prompt.md`）中选取与检测类型匹配的 prompt。
        2.  通过指定的基准渠道请求大模型，获取其输出。
        3.  将此输出作为“黄金标准”存入 `model_baselines` 表。

2.  **基准更新**:
    *   管理员可以随时为某个模型重新生成或更新基准。更新操作会覆盖旧的基准数据。

### 4.3 自动化监控流程

1.  **调度**: 一个后台任务（如每小时）扫描所有配置了 `monitoring_config` 的渠道。
2.  **探测**:
    *   对于每个待测渠道，根据其配置的 `target_model`, `test_type` 等参数，从 `model_baselines` 表中加载对应的基准数据和 prompt。
    *   向待测渠道发送该 prompt。
3.  **评估与打分**:
    *   **简单对比**: 对于 `encoding` 或 `structure_consistency` 等检测类型，可以通过固定的规则（如代码单元测试、JSON Schema校验）直接判断输出是否正确，并给出 0 或 1 的分数。
    *   **复杂对比 (裁判LLM)**: 对于 `style`, `reasoning` 等主观类型，系统会将**基准输出**和**待测输出**一同发送给一个高质量的“裁判LLM”（如 GPT-4o 或 Claude 3 Opus），并要求其根据评估标准（宽松/标准/严格）给出一个差异分数（如 0-100）。
4.  **结果存储**:
    *   将每次的探测结果，包括原始输出、差异分数、是否通过等信息，存入 `model_monitoring_results` 表。

### 4.4 API 设计

*   **基准管理**:
    *   `POST /api/models/baselines`: 创建或更新一个模型的基准。
        *   **Request Body**: `{ "model_name": "gpt-4", "baseline_channel_id": 123, "test_type": "style", "evaluation_standard": "strict" }`
    *   `GET /api/models/baselines`: 获取所有模型的基准配置。

*   **监控结果查询**:
    *   `GET /api/channels/:id/monitoring`: 获取指定渠道的监控历史。
        *   **Query Params**: `model_name`, `test_type`, `start_time`, `end_time`
    *   `GET /api/models/:model_name/monitoring_report`: 获取一个模型在所有渠道下的最新监控对比报告。
        *   **响应示例**:
            ```json
            {
              "model_name": "gpt-4",
              "baseline_channel_id": 123,
              "last_test_time": 1733433900,
              "results": [
                {
                  "channel_id": 124,
                  "channel_name": "Channel-B",
                  "diff_score": 95.5,
                  "is_pass": true
                },
                {
                  "channel_id": 125,
                  "channel_name": "Channel-C",
                  "diff_score": 78.0,
                  "is_pass": false,
                  "failure_reason": "Style imitation failed: verbosity mismatch."
                }
              ]
            }
            ```

## 5. P2P共享分组统计 (P2P Group Statistics)

本章节详细设计P2P共享分组的聚合统计方案，以满足需求 3.1, 3.2。

### 5.1 聚合逻辑

P2P分组的统计数据是其**组内所有渠道**同类指标的逻辑汇总。

*   **求和类指标**:
    *   `TPM`, `RPM`, `quota_pm`, `total_tokens`, `total_quota`, `total_sessions`, `unique_users`
    *   **计算**: `Group.TPM = SUM(Channel.TPM) for all channels in group`

*   **平均类指标**:
    *   `avg_concurrency`, `downtime_percentage`
    *   **计算**:
        *   `Group.avg_concurrency` = `SUM(Channel.avg_concurrency)` (并发数是叠加的)
        *   `Group.downtime_percentage` = `AVG(Channel.downtime_percentage)` (取组内渠道的平均停机时间占比)

*   **聚合范围**:
    *   可以按分组下的**特定模型**进行聚合（如 "TeamA" 分组下 "gpt-4" 的总TPM）。
    *   也可以按分组的**全部模型**进行聚合（`model_name = '_all'`）。

### 5.2 触发与更新机制

分组的统计更新**不**采用高频的实时计算，而是依赖于渠道统计数据的持久化过程，以降低系统负载。

1.  **触发时机**:
    *   在 **L3 - 数据库持久化** 任务（每15分钟执行一次）中，当一个渠道的统计数据被成功更新到 `channels` 表后，系统会检查该渠道所属的所有P2P分组。

2.  **更新检查 (超时机制)**:
    *   对每个关联的P2P分组，检查其在 `group_stats` 表中的 `updated_at` 时间戳。
    *   如果 `当前时间 - updated_at > 30分钟`（半小时），则认为该分组的统计数据已“过时”，触发一次更新。

3.  **更新过程**:
    *   **锁定与并发控制**:
        *   为了防止多个渠道的更新操作同时触发同一个分组的统计，引入一个分布式锁（基于Redis `SETNX`）。
        *   在更新前尝试获取 `group_stats_lock:{group_id}` 锁。
        *   同时，限制整个平台并发执行分组统计的协程数量（如最多5个），避免数据库I/O风暴。
    *   **数据聚合**:
        1.  获取该分组下的所有渠道ID。
        2.  从 `channels` 表中批量查询这些渠道的最新统计数据。
        3.  根据 **5.1节** 的聚合逻辑，在内存中计算出分组的总计指标。
        4.  将结果`UPSERT`到 `group_stats` 表中，并更新 `updated_at` 时间戳。
    *   **释放锁**: 更新完成后，释放分布式锁。

### 5.3 API 设计

*   **Endpoint**: `GET /api/p2p_groups/:id/stats`
*   **权限**: 分组内的成员
*   **参数**:
    *   `model_name` (Query, string, optional): 指定要查询的模型，如果为空则返回该分组下所有模型的聚合统计（`model_name = '_all'`）。
*   **响应**:
    *   直接从 `group_stats` 表中查询并返回数据。
    ```json
    {
      "group_id": 456,
      "model_name": "gpt-4", // or "_all"
      "tpm": 500000,
      "rpm": 120,
      "quota_pm": 300000,
      // ... other stats fields
      "updated_at": 1733434200
    }
    ```
